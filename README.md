# Mailwyrm

Mailwyrm is an AI-first email client for Gmail. Its core job is to protect human attention by separating human correspondence from machine correspondence, then turning routine machine mail into a daily digest with links back to the original messages.

The project should treat Gmail as the source of truth. Mailwyrm is not a separate mailbox, a forwarding service, or a parallel archive. Actions taken in Mailwyrm should be reflected in Gmail, and actions taken in Gmail should be reflected back in Mailwyrm.

## Product Intent

Most email clients organize messages. Mailwyrm should organize attention.

The first strong product promise is:

> Your inbox should mostly contain mail from humans, not machines.

Human correspondence includes personal messages, direct replies, work conversations, and small-group threads where a person likely expects a response.

Machine correspondence includes receipts, shipping notices, newsletters, alerts, account notifications, automated confirmations, terms updates, SaaS notifications, and other mail generated by systems.

Machine mail is not automatically worthless. Some machine mail is important, sensitive, or time-critical. Mailwyrm should classify it, summarize it, and apply user-approved policy with clear auditability.

## Initial MVP

1. Connect to Gmail with the minimum scopes needed to read and modify mailbox state.
2. Build a local index of Gmail messages, threads, labels, and sync cursors.
3. Classify incoming mail as human, machine, or needs review.
4. Apply Gmail-visible labels for classification.
5. Generate a daily machine-mail digest with links to original messages.
6. Allow user-approved automation for archive and trash behavior.
7. Keep an audit trail of AI decisions and mailbox actions.

## Product Principles

- Gmail remains authoritative.
- Human mail deserves foreground attention.
- Machine mail should become structured background information.
- Automation should be confidence-based and reversible where possible.
- Destructive actions require explicit user trust, narrow policy, and audit logs.
- The user should always be able to understand why an action happened.
